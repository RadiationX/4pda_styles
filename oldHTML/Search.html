СТРУКТУРА ЖУРНАЛА
(Все файлы должны иметь кодировку utf-8 БЕЗ BOM)
folder/sys/main.txt - Главное меню журнала
folder/0.html - Откроется при клике по первому пункту (главного меню журнала)
folder/1.html - Откроется при клике по второму пункту
folder/10.html - Откроется при клике по девятому пункту

Файлы меню:
Должен содержать в себе список пунктов, каждый пункт с новой строки. (желательно *.txt)

Файлы журнала:
Любой текст с поддержкой html, css, javascript. (*. html)

Для перехода на новое меню ссылайтесь на файл со следующим содержанием:
menu
help
help/main.txt


menu- означает что нужно запустить окно со списком
help- папка с файлами журнала (0. html,1. html,...,10. html)
help/main.txt - путь до файла с меню

Если вы создали файл с произвольным названием, к примеру help. html, то что-бы его открыть нужно разместить html ссылку:
<a href="file:///android_asset/help. html">текст ссылки</a>

Искал по форуму, ничего похожего не нашел, поэтому напишу вот такую информацию, может, кому пригодится:

О виджетах бады
Собстно в данный момент занимаюсь разработкой под баду, и недавно столкнулся с виджетами. До этого виджеты видел в Опере на ББ и на Андроиде. Так вот, насчет виджетов Оперы - они волне себе легко (для понимающего в html/ css/js) портируются на баду. Для этого ставится:
1) Эклипс хелиос 3.6
2) в него доставляется плагин от самсунга для разработки виджетов под их девайсы (Help - Add new Software...)
http://widget.samsungmobile.com/sdk/
3) немного читается раздел приложенного хелпа, о структуре проекта и назначениях папок, создается проект виджета
4) инсталлится понравившийся виджет оперы, в настройках при инсталляции запоминается его папка, после инсталляции переходим в саму папку (обычно диск:\Documents and Settings\юзер\Local Settings\Application Data\имя виджета)

Оттуда забираем папки с js, картинками, css, index. html, и, возможно, какие-то файлы из корня. Не нужен config.xml, install.conf, uninstall, *.log, *.exe. Переносим в проект виджета, собираем, отлаживаем, портируем, жмем Widget->Package (Ctrl+Shift+P по умолчнаию), получаем wgt файл, копируем на телефон, там открываем файловым менеджером и инсталлим.

Другой вариант - попробовать проинсталлить сразу wgt файл оперного виджета. В зависимотси от функционала и размера экрана, может встать и совсем без проблем.

Работоспособность ессно не гарантирована, при втором способе. Доступа к специфическим функциям телефона тоже по умолчанию не будет (надо дописывать). Но всякие часики-красивости-rss читалки можно поставить почти без проблем, и с минимальными трудозатратами (измениния в css и свойствах проекта) адаптировать под мобилку.

Но с любительскими туго, после Joomla вообще труба. ИМХО если нет задачи разработать что-то принципиально новое (или просто своё), то смысла в modx немного.

Ну и, допустим, как таковых готовых шаблонов оформления для modx нет (по крайней мере, год назад не было). Это объясняется тем, что modx по сути работает с чистым html- css с минимальными (и весьма удобными) вставками собственно самой логики modx (и это действительно просто и хорошо работает). Но проблема в том, что в последние годы студии гораздо более заинтересованы именно в создании шаблонов под конкретные движки на собственных фреймворках, все это красиво сделано, собрано в многочисленных коллекциях и хорошо работает. Тогда как на чистом HTML/ CSS всегда попадается всякая хрень, которая смотрится, мягко говоря, не очень. Зато потом легко редактировать.

Лично я отказался от modx и перешел на Joomla именно из-за оформления: денег на дизайнера у меня не было, в Интернете попадалась всякая хрень, а готовые шаблоны для Joomla выглядели просто замечательно.

Ну и плюс если вдруг планируется регулярное обновление, то тем более не советую. Он больше для "постоянных" сайтов.

Короче, хз...я бы не стал вот так переходить к modx без реальной необходимости. Скорее всего, если вопрос повис в воздухе, то и необходимости нет. Скорее всего, через 6-10 месяцев захочется вернуться к Joomla.

З.Ы. Но вообще надо отдать должное modx'у. С течением времени он не становился монстром, как это произошло с Wordpress или Joomla. Строгое разделение внешнего вида, структуры и программной логики при должном навыке дает потрясающие возможности для создания и редактирования вообще всего. А вот Joomla скатилась в какую-то жопу, редактировать готовые решения стало практически невозможно, создавать новые также становится все сложнее. И даже разделение файловой структуры на директории выглядит как-то по-издевательски: на кой они нужны, эти директории, если файлы, отвечающие за внешний вид страниц, порой можно найти ГДЕ УГОДНО, ТОЛЬКО НЕ в /templates? В последнее время для поиска необходимого файла с каким-нибудь CSS-свойством я заходил в консоль Linux на сервере и запускал поиск по содержимому файлов, иначе я бы никогда эти свойства не нашел. В чем смысл клепания новых версий без обратной совместимости? Для меня это все очень непонятно. Такое ощущение, что всем своим сознанием сообщество разработчиков застряло где-то в 90-х.
